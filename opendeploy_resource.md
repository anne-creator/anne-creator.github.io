## 1. Executive Summary

OpenDeploy is an agent-native deployment and production control plane for the vibe coding era. Traditional tools such as Vercel, Supabase, Railway, Render, Fly.io, and cloud dashboards were designed for human developers who read docs, configure settings, upload code, and click through dashboards. OpenDeploy starts from a different premise: the primary operator is the AI coding agent, and the human stays in the loop for observation, approval, billing, and accountability.

OpenDeploy lets a user tell an agent to deploy a project, after which the agent analyzes the codebase, detects framework and environment needs, provisions dependencies, deploys the app, watches logs, and returns a live URL. The long-term vision is not only "deploy this" but a complete production loop: deploy agent -> runtime agent -> oncall agent -> security agent.

As of June 1, 2026, after one week of beta testing, OpenDeploy had 200+ free users and 3 paid users, including 1 paid user in the United States. This traction signal is user-reported and should be treated as an internal metric until updated with billing/dashboard evidence.

## 5. Problem

AI coding agents are changing the software production loop. A user can ask an agent to build an app, edit files, run tests, and fix bugs, but the last mile still breaks on traditional DevOps surfaces.

### Deploy Problem

Agents can write code, but deployment platforms still assume a human operator. Vercel, Railway, Render, Fly.io, AWS, and similar systems typically rely on login flows, dashboard configuration, manual env handling, and human interpretation of build failures. Agents need a stable CLI/API/skill deploy target that can analyze the project, provision what is needed, and return a live URL.

### Runtime Problem

Deployment is only the start. Production systems include service graphs, resource quotas, health checks, costs, dependency status, clusters, regions, and scaling rules. Traditional platforms show those concepts to humans but do not consistently expose them as agent-readable runtime primitives.

### Oncall Problem

Production services fail: build failures, startup failures, slow responses, quota issues, dependency errors, domain/SSL problems, and regressions. Logs and metrics are still primarily designed for human operators. Agents need a loop that can read logs and metrics, explain root cause, suggest or apply safe fixes, retry, roll back, and notify humans when approval is needed.

### Security Problem

Letting agents touch production raises permission and compliance risk. Personal cloud keys, broad credentials, production secrets, and unscoped service access are dangerous. The market needs agent-specific temporary credentials, narrow route permissions, secret upload consent, audit trails, leak detection, vulnerability scanning, risky deploy blocking, and human approval gates.

### Core Thesis

Agents can already build software, but they do not yet have their own production runtime.

## 6. Solution

OpenDeploy is a standard production path for agents:

1. The user asks the agent to deploy.
2. The agent installs or invokes OpenDeploy through a skill, CLI, prompt handoff, or API.
3. OpenDeploy analyzes the project.
4. OpenDeploy builds and deploys the app.
5. Dependencies are provisioned and env vars are handled through consented flows.
6. Logs, status, and deployment history become readable by both agent and human.
7. The agent returns a live URL.
8. The human verifies, binds the project, upgrades, configures domain, or approves risky actions.

OpenDeploy is not only a PaaS with an AI integration. It is an agent-first deployment and observability control plane.

## 7. Roadmap Narrative

OpenDeploy should be described as a four-stage production loop:

| Stage | Capability | Goal |
| --- | --- | --- |
| Deploy agent | Framework/env detection, one-command deploy, live URL return | Let agents safely ship apps |
| Runtime agent | Service graph, health, autoscaling, cost optimization | Let agents understand production state |
| Oncall agent | Logs, metrics, failure notifications, root cause, fix and retry | Let agents handle first-line production failure |
| Security agent | Secret leak detection, vulnerable dependency scan, unsafe config patching, risky deploy blocking | Let agents enter production with governance |

Public messaging should avoid claiming all four stages are fully shipped. The stronger narrative is: deploy agent is the wedge, runtime/oncall/security are the compounding roadmap.

## 8. Technical Architecture

### Agent Access Layer

- OpenDeploy skills and plugins define the standard agent workflow.
- `/deploy` or `/opendeploy` acts as the natural command interface.
- CLI and npm binaries support local terminal, CI, and agent shell workflows.
- Agent launcher on the marketing site lets users copy prompts or open their agent.
- Local auth/context files preserve credentials and project/service/deployment IDs for repeat deploys.

### Gateway And Security Model

- Gateway is the external API entry point for OIDC login, API keys, guest credentials, rate limits, quotas, circuit breakers, and reverse proxy routing.
- Guest credential gives a user no-account first deploy through narrow routes.
- Account binding transfers guest project ownership into a real account.
- Env upload requires per-project consent and should expose keys, not secret values, to the agent/human UI.
- User API keys support CLI, CI/CD, IDE plugins, and long-lived agent integrations.

### Build And Deployment System

- Build service analyzes code, builds images, pushes images, detects dependencies, and can generate Dockerfile/Compose/CI plans.
- Deployment service orchestrates create, stop, start, restart, cancel, retry, promote, and rollback.
- Analyzer service uses LLM-assisted framework detection, dependency detection, build error diagnosis, and deploy planning.
- Cluster service manages Kubernetes clusters, regions, worker agents, and agent pools.
- Worker agent performs Docker/Kubernetes actions and streams logs/status back.

### Observability, Billing, And Operations

- Log capture can read pod stdout and write to a queryable store such as ClickHouse.
- Monitoring aggregates Kubernetes, Prometheus, and log-derived signals for health, quota, and alerting.
- Notification service sends email, webhook, Discord, Slack, DingTalk, WeCom, or similar alerts.
- Billing service supports subscriptions, credits, AI Hub usage, Stripe webhooks, affiliate programs, and Enterprise plans.
- Training/data service captures build, deployment, LLM interaction, log, and metrics patterns to improve future diagnosis and safety.

## 12. Competitive Landscape

The competitive story should be precise. Railway and Zeabur have both released agent-facing experiences, so OpenDeploy should not claim "only platform with agent support." The stronger claim is that OpenDeploy is agent-native from first principles.

### Railway

Railway has an Agent Directory, Agent Skills, MCP/Remote MCP options, CLI paths, dashboard agent, logs, metrics, templates, rollback, team collaboration, OAuth, and short-lived tokens.

Railway's strength: mature PaaS, trusted infra, brand, templates, team workflows, and strong developer adoption.

OpenDeploy's distinction: Railway is an agent-enabled PaaS. OpenDeploy aims to be an agent-native production control plane where guest credential, narrow routes, deploy chain, observe/fix loop, and human approval gates are designed around agents from the beginning.

### Zeabur

Zeabur Agent Skills position Zeabur as an AI DevOps surface that lets Claude Code deploy services, rent servers, manage configs, handle AI, email, domain, env vars, ports, health checks, firewalls, containers, databases, and troubleshooting.

Zeabur's strength: strong Chinese/Asia developer awareness, VPS/BYOS, domain/email/AI Hub bundling, and very direct AI DevOps narrative.

OpenDeploy's distinction: Zeabur feels like an agent operating a broad cloud platform. OpenDeploy should feel like a deploy + observe + safety control plane designed for agents and humans sharing production responsibility.

### Vercel / Netlify

Strength: polished frontend deployment, templates, preview links, brand, and ecosystem.

Gap: more human/dashboard oriented, stronger on frontend than full-stack production agent workflow, less centered on agent credentials and observe/fix loop.

OpenDeploy distinction: agent-native deploy for full-stack projects, dependencies, credentials, live URL, logs, and safety gates.

### Supabase / InsForge

Strength: backend-as-a-service, databases, auth, storage, functions, developer-friendly APIs.

Gap: not primarily a general deploy target for arbitrary agent-generated full-stack apps; not the production runtime for the agent itself.

OpenDeploy distinction: deploys and runs the app; can integrate with managed dependencies or other BaaS tools while preserving agent-first deploy/observe workflow.

### Render / Fly.io / AWS / GCP / Azure

Strength: mature infrastructure, broad primitives, enterprise trust, flexible compute.

Gap: configuration and operational complexity; direct agent access can be risky without narrow credentials and human approval gates.

OpenDeploy distinction: wraps production primitives into agent-safe actions.

### Agent Frameworks And IDEs

Cursor, Claude Code, Codex, OpenCode, LangGraph, CrewAI, AutoGen, E2B, and Daytona help agents write, run, or coordinate work.

Gap: many do not solve durable production hosting, logs, runtime state, rollback, costs, and security gates end to end.

OpenDeploy distinction: becomes the deploy/observe backend for agent-built software.
